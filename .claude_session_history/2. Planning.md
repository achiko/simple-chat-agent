     Action Plan — Next.js AI Chat Agent (V1.1 Full)

     Derived from prd-draft.md. Single-source build roadmap for the hybrid
     text-streaming + async image-generation platform.

     1. Objective

     Build a full-stack AI platform that:
     - Accepts chat-style prompts (Text or Image).
     - Processes every request as a job through a BullMQ queue + worker.
     - Streams text token-by-token over SSE, fed by Redis Pub/Sub.
     - Generates images asynchronously via OpenAI, surfacing them in a gallery.
     - Persists full job history, token usage, and cost in Postgres.
     - Exposes a live System dashboard for queue, worker, and stream
     observability.

     MVP excludes auth and multi-provider support (per PRD §15).

     2. Confirmed Stack

     ┌────────────────┬────────────────────────────────────────────────────────┐
     │     Layer      │                         Choice                         │
     ├────────────────┼────────────────────────────────────────────────────────┤
     │ Framework      │ Next.js 15 (App Router) + TypeScript                   │
     ├────────────────┼────────────────────────────────────────────────────────┤
     │ Styling / UI   │ Tailwind CSS + shadcn/ui                               │
     ├────────────────┼────────────────────────────────────────────────────────┤
     │ ORM / DB       │ Prisma + PostgreSQL 16                                 │
     ├────────────────┼────────────────────────────────────────────────────────┤
     │ Queue /        │ BullMQ + Redis 7 (same Redis for both)                 │
     │ Pub/Sub        │                                                        │
     ├────────────────┼────────────────────────────────────────────────────────┤
     │ AI provider    │ OpenAI SDK — gpt-5 (text streaming), gpt-image-1       │
     │                │ (images)                                               │
     ├────────────────┼────────────────────────────────────────────────────────┤
     │ Tests          │ Playwright (E2E)                                       │
     ├────────────────┼────────────────────────────────────────────────────────┤
     │ Runtime        │ Node 20+, pnpm                                         │
     ├────────────────┼────────────────────────────────────────────────────────┤
     │ Orchestration  │ Docker Compose (app, worker, redis, postgres)          │
     └────────────────┴────────────────────────────────────────────────────────┘

     3. Architecture

     ┌────────┐   POST /api/jobs   ┌────────┐   BullMQ   ┌────────┐   OpenAI
     │ Client │ ─────────────────▶ │  API   │ ─────────▶ │ Worker │ ─────────▶
     │ (React)│                    │ Next.js│            │  Node  │
     └────┬───┘                    └────┬───┘            └───┬────┘
          │ SSE /api/jobs/:id/stream    │                    │
          │◀────────────────────────────┘◀────── Redis ──────┘
                                   subscribe      Pub/Sub
                                                   ▲ publish chunks
                                     Postgres  ◀───┴─── final result, usage,
     cost

     - Text flow: Frontend → API → Queue → Worker → OpenAI streaming → Redis
     PUBLISH job:{id}:stream → API SSE → UI; final output, tokens, cost → DB.
     - Image flow: Frontend → API → Queue → Worker → OpenAI images → DB → UI
     (polled or status-only SSE).

     4. Data Model (Prisma)

     jobs
     - id (cuid) · prompt (text) · type enum TEXT | IMAGE
     - status enum PENDING | QUEUED | STARTED | STREAMING | COMPLETED | FAILED |
      CANCELLED
     - input_tokens, output_tokens, total_tokens (int, nullable until
     completion)
     - estimated_cost (decimal) · model (string) · error (text, nullable)
     - created_at, updated_at, started_at, completed_at

     results
     - id · job_id FK → jobs.id (unique) · output (text — token stream for text,
      URL/b64 for image)

     6. Phased Build

     Each phase lists Goal → Tasks → Exit criteria. Phases are ordered so each
     builds a runnable slice.

     Phase 0 — Scaffold (½ day)

     - Goal: boot all four services locally.
     - Tasks:
       - pnpm create next-app (TS, App Router, Tailwind, src/).
       - pnpm dlx shadcn@latest init; add base components (button, input, tabs,
     card, badge).
       - Add .env.example (DATABASE_URL, REDIS_URL, OPENAI_API_KEY, pricing
     vars).
       - Write docker-compose.yml with app, worker, redis:7, postgres:16.
       - Dockerfile (multi-stage Next build) + worker.Dockerfile.
     - Exit: docker compose up → Next.js default page at :3000, worker container
      idle, Redis + Postgres healthy.

     Phase 1 — Prisma schema + migrations (½ day)

     - Goal: persistent jobs + results.
     - Tasks:
       - Define prisma/schema.prisma per §4.
       - prisma migrate dev --name init.
       - src/lib/db.ts Prisma singleton.
     - Exit: migrations applied; Prisma Client generated; SELECT on jobs works.

     Phase 2 — Queue + worker skeleton (1 day)

     - Goal: end-to-end lifecycle transition on a no-op job.
     - Tasks:
       - src/lib/queue.ts — BullMQ Queue + QueueEvents; typed job payload { 
     jobId: string }.
       - worker/index.ts — BullMQ Worker, concurrency = 4.
       - worker/heartbeat.ts — every 5s write SET worker:heartbeat <ts> EX 15.
       - Retry config: attempts: 3, backoff: { type: 'exponential', delay: 1000 
     }.
       - Status helper setStatus(jobId, status, patch?).
     - Exit: enqueued test job walks PENDING → QUEUED → STARTED → COMPLETED.

     Phase 3 — Core API routes (½ day)

     - Goal: create / read jobs via HTTP.
     - Tasks: implement POST /api/jobs, GET /api/jobs, GET /api/jobs/:id.
     - Exit: curl round-trips succeed; job row created and enqueued.

     Phase 4 — Text streaming (2 days — critical path)

     - Goal: token-by-token UI from OpenAI.
     - Tasks:
       - Worker processors/text.ts: openai.responses.stream({ model: 'gpt-5', 
     input: prompt }), iterate deltas.
       - For each delta: RPUSH job:{id}:chunks <delta> and PUBLISH 
     job:{id}:stream <delta> (list backs reconnect replay).
       - On response.completed: persist full text to results, write usage
     (input/output tokens) + estimated_cost, set status COMPLETED, publish
     {done:true}.
       - GET /api/jobs/:id/stream (SSE): on connect, LRANGE job:{id}:chunks →
     replay, then subscribe to job:{id}:stream; close on terminal status.
       - Client hook useJobStream(id) with EventSource, auto-reconnect on error.
       - Chat UI streaming bubble.
     - Exit: submit text prompt → chunks render as they arrive; close + reopen
     tab mid-stream → replay completes without data loss.

     Phase 5 — Image generation (1 day)

     - Goal: async image job visible in gallery.
     - Tasks:
       - Worker processors/image.ts: openai.images.generate({ model: 
     'gpt-image-1', prompt, size: '1024x1024' }).
       - Store returned URL or base64 in results.output, set COMPLETED.
       - Client polls GET /api/jobs/:id every 2s until terminal (or uses
     status-only SSE variant).
     - Exit: image prompt → gallery card shows final image plus prompt and cost.

     Phase 6 — Token tracking + cost (½ day)

     - Goal: every completed job carries tokens + cost.
     - Tasks:
       - src/lib/pricing.ts: per-model price map (input/output per 1M tokens)
     sourced from env; gpt-image-1 per-image price by size.
       - Always read usage from the OpenAI response — never estimate (mitigates
     PRD §14 token-miscalc risk).
       - Compute estimated_cost = input_tokens * input_price + output_tokens * 
     output_price; write inside the same Prisma transaction that flips status to
      COMPLETED.
     - Exit: no COMPLETED row has null tokens or cost.

     Phase 7 — Frontend tabs (1.5 days)

     - Goal: the four PRD tabs are usable.
     - Tabs:
       - Chat — prompt textarea, Text | Image selector, submit; live streaming
     bubble (text) or spinner→image (image); show tokens + cost on completion.
       - History — reverse-chronological list, filter by type / status; red
     state for FAILED with error text.
       - Gallery — grid of COMPLETED image jobs (prompt, cost).
       - System — see Phase 8.
     - Exit: all four tabs functional against real backend.

     Phase 8 — System dashboard (1 day)

     - Goal: live observability.
     - Tasks:
       - GET /api/system/stats returns:
           - Queue counts via queue.getJobCounts('waiting', 'active', 
     'completed', 'failed').
         - Worker online/offline from worker:heartbeat TTL.
         - Active stream count: maintain an in-memory counter in the API
     process, incremented on SSE connect / decremented on disconnect.
         - Last 50 log lines: worker pushes to a capped Redis list (LPUSH +
     LTRIM 0 49).
       - Dashboard tab polls /api/system/stats every 2s.
     - Exit: enqueue 10 jobs → waiting climbs and drains in real time; log tail
     scrolls.

     Phase 9 — Retry + error handling (½ day)

     - Goal: failures are persistent and visible.
     - Tasks:
       - BullMQ retry already configured in Phase 2; on final failure worker
     sets FAILED and writes error.
       - UI surfaces failures in Chat (red bubble) and History.
     - Exit: forced OpenAI failure shows 3 retries in logs, then FAILED row with
      error text in History.

     Phase 10 — Playwright E2E (1 day)

     - tests/chat.spec.ts — submit text prompt, assert streaming deltas arrive,
     final tokens + cost render.
     - tests/image.spec.ts — submit image prompt, wait for completion, assert
     gallery card.
     - tests/history.spec.ts — forced failure appears with error.
     - tests/dashboard.spec.ts — enqueue 5 jobs, assert counters update.
     - CI script: pnpm test:e2e running against docker compose up.
     - Exit: all four specs green.

     8. Risks (PRD §14) and mitigations

     ┌─────────────────────┬───────────────────────────────────────────────────┐
     │        Risk         │                    Mitigation                     │
     ├─────────────────────┼───────────────────────────────────────────────────┤
     │ Streaming           │ Worker appends every delta to job:{id}:chunks     │
     │ disconnect / tab    │ Redis list; SSE endpoint replays the list on      │
     │ reload              │ connect before subscribing to Pub/Sub.            │
     ├─────────────────────┼───────────────────────────────────────────────────┤
     │ Queue overload      │ Worker concurrency cap + BullMQ rate limiter;     │
     │                     │ dashboard surfaces waiting backlog early.         │
     ├─────────────────────┼───────────────────────────────────────────────────┤
     │ Token               │ Always read usage from OpenAI response; never     │
     │ miscalculation      │ estimate from prompt length.                      │
     ├─────────────────────┼───────────────────────────────────────────────────┤
     │ Redis Pub/Sub       │ Chunks list (append-only) is the source of truth; │
     │ message loss        │  Pub/Sub is a low-latency fanout hint.            │
     └─────────────────────┴───────────────────────────────────────────────────┘
