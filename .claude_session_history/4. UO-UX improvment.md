     Improvements Plan ‚ÄĒ Chat Sessions, Sidebar, Thinking State, Title, Status

     Context

     The chat UI at chat-ui/ treats each prompt as an isolated Job row. Users
     cannot revisit prior conversations: messages live only in useState and
     vanish on refresh. There is no concept of a session, no way to jump back to
      a specific chat, and no distinction between a request that's still
     thinking and one that has produced an empty response. The System dashboard
     shows aggregate counts but no way to drill into the actual Completed/Failed
      jobs. The browser tab still reads "Next.js Chatbot Template" from the
     template.

     This plan introduces a ChatSession model, a routed chat page
     (/chat/[sessionId]), a chat-only sidebar for navigation, an animated
     thinking indicator, a corrected page title, and clickable Completed/Failed
     filters on the System tab. User-confirmed scope: sessions group both TEXT
     and IMAGE jobs; sidebar is chat-only (not global); System filters operate
     on jobs; History tab stays alongside the sidebar.

     Schema & Migration

     File: lib/db/schema.ts

     1. Add a new chatSession table after jobs:
       - id uuid pk defaultRandom()
       - userId uuid NOT NULL references user.id
       - title text NOT NULL (derived from first prompt ‚ÄĒ first 60 chars,
     trimmed)
       - createdAt timestamp NOT NULL defaultNow()
       - updatedAt timestamp NOT NULL defaultNow() (bumped on each new job so
     sidebar sorts by latest activity)
       - Index on (userId, updatedAt desc).
     2. Extend jobs with sessionId uuid NULL references chatSession.id ON DELETE
      SET NULL. Nullable so legacy rows survive without backfill. Index
     (sessionId, createdAt asc) for rehydration order.
     3. Run pnpm db:generate ‚Üí emits lib/db/migrations/0002_*.sql. Apply locally
      via pnpm db:migrate. Compose's migrate service already runs this at stack
     boot.

     Why SET NULL not CASCADE: deleting a session shouldn't erase
     billing/history artifacts. Orphaned jobs remain visible in the History tab.

     Risks / Edge Cases

     - Guest isolation ‚ÄĒ every session endpoint AND the sessionId path in POST 
     /api/jobs must compare session.userId to auth().user.id. Without this, any
     guest can read another guest's history by guessing UUIDs. This is the
     single most important thing to code-review.
     - Mid-stream navigation ‚ÄĒ on rehydrate, re-open SSE for any job still in
     STREAMING/STARTED/QUEUED; otherwise the UI shows an empty assistant bubble
     forever.
     - Session ordering ‚ÄĒ sort by updatedAt desc (not createdAt); a long
     conversation stays on top.
     - / for returning users ‚ÄĒ always renders a fresh chat; never auto-redirects
      to the latest session (would violate "New chat" mental model).
     - Drizzle + postgres Date ‚ÄĒ use .set({ updatedAt: new Date() }); never
     interpolate a Date into sql\``.
     - cacheComponents: true ‚ÄĒ no export const dynamic in new routes; rely on
     await params and auth() to implicitly mark them dynamic.
     - Double-click on Send ‚ÄĒ creatingSessionRef guards against two parallel
     POST /api/sessions calls creating duplicate sessions.
     - Session title quality ‚ÄĒ simple truncation of the first prompt. AI
     summarisation is a tempting upgrade but adds a second queue hop and spend;
     skip for v1.
